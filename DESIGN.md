### 1. 데이터 분석

- metadata.json 해석
  metadata.json은 단순한 파일 목록이 아니라 다음과 같은 계층적 도면 관계를 가지고 있다고 해석하였습니다.

```
프로젝트
 └─ 공간 (Space)
     └─ 공종 (Discipline)
         └─ 영역 (Region, optional)
             └─ 리비전 (Revision)
```

도면은 단순히 이미지 파일이 아니라서로 상위-하위 관계를 가지며 이 데이터는 단순한 구조가 아니라 좌표계가 연결된 도면 네트워크에 가깝다고 해석하였습니다.

- 데이터 구조 이해
  데잉터 구조에 대한 이해는 제공된 README.md 파일을 통해 공간, 공종, 영역, 리비전에 대해 이해하고 주어진 주요 필드 설명을 통해
  type을 정의하고 필드마다 부여된 설명을 통해 어떠한 데이터를 가지고 있는지 이해하였습니다.

### 2.접근 방식
- 과제 접근 순서 및 방식
  과제를 시작하기 전에 우선 metadata.json과 README를 통해 도메인 이해에 집중했습니다.
  건설 도면 용어(공종, 리비전, 평면도 등)에 대한 이해가 부족했기 때문에 개념을 먼저 정리한 뒤 구현을 진행했습니다.
  이후 사용자 시나리오를 분석하며 도면 선택 및 조회 기능을 포함해 리비전 간 비교공종 간 도면 비교 기능이 필요하다고 판단했습니다.
  구현은 다음 순서로 진행했습니다. 복잡한 데이터 구조를 평탄화하여 index 형태로 정리하고 공간–공종–영역–리비전 탐색 흐름 구현,
  선택된 도면 표시 및 현재 상태 인지 UI 구성, 동일 공종 내 Base/Overlay 비교 기능 추가 공종 간 비교가 가능하도록 Base 축과 Overlay 축을 분리 설계
  필수 기능을 구현하고 추가적인 기능을 확장하며 초기 설계에서 시작해 사용자 시나리오에 맞춰 비교 구조를 발전시키는 방식으로 접근했습니다.

### 3.UI 설계 결정
  선택한 레이아웃
  ```
  [ Sidebar ] | [ Viewer ] | [ RightPanel ]
  ```
  좌측: 탐색 축 (공간/공종/영역)
  중앙: 도면 표시
  우측: 리비전 및 비교 설정
  #### 왜 이 방식이 최선이라고 판단했는가 ?
  과제를 받고 시나리오 및 요구사항을 보고 도면이 중심 및 중점이 되어야 된다고 판단하였고 탐색은 좌측에서 비교 조작은 우측에서 기능 영역을 컨텍스트 물리적으로 분리하여
  시선 흐름이 중앙에 머물도록 설계했습니다. 또한 색상도 고려하여 가운데에 집중이 되도록 연한 그레이색을 양 옆에 배치하여
  중앙에 머물도록 하였습니다. 또한 viewer 부분을 카드 형식으로 하여 조금 더 시선이 집중되도록 구현하였습니다.

  #### 고려했던 대안
  도면이 중점 및 중심이 되고 탐색 및 조작 panel의 위치를 현재와 다르게 짠 layout도 대안에 있었습니다.
  대안과 현재 layout을 비교해 현재 layout이 사용자 입장에서 조금 더 편리할 것 이라고 판단하였습니다.

### 4.기술 선택
  React + TypeScript
  - 상태 기반 UI 구조에 적합하여 사용
  - 도면, 공종, 리비전 등 계층적 데이터를 타입으로 명확히 정의
  - 타입 안정성 유지

  Zustand
  - Base와 Overlay 축이 완전히 독립된 상태 구조 필요
  - Sidebar, Viewer, RightPanel 간 상태 공유 필요
  - Context 대비 보일러플레이트가 적고 구조가 단순

  Tailwind CSS + clsx
  - 빠른 UI 반복 설계 가능
  - 조건부 스타일링을 clsx로 간결하게 표현 가능

### 5.어려웠던 점

  - 복잡한 JSON 구조 해석 및 평탄화
    metadata.json은 단순한 트리 구조가 아니라 공간 → 공종 → 영역 → 리비전의 계층 구조region이 존재하는 공종과 그렇지 않은 공종의 차이가 혼재되어 있어 구조 이해에 시간이 필요했습니다.렌더링 단계에서 매번 깊은 계층을 탐색하는 것은 비효율적이라고 판단하여 key 기반으로 평탄화하여 index를 구성하는 과정이 가장 고민이 많았던 부분입니다. 데이터를 단순 조회 구조로 재정렬하는 과정이 이번 과제의 기술적 난이도 중 가장 큰 부분이었습니다.

  - 사용자 편의성을 고려한 UI/UX 설계
    단순히 기능을 구현하는 것이 아니라 현재 어떤 도면을 보고 있는지 인지 가능해야 하고 비교 상태(Base/Overlay)를 사용자가 명확히 인지해야 하며
    중앙 도면이 항상 시각적 중심이 되어야 한다 는 점을 동시에 만족시키는 UI 구성이 어려웠습니다.
    도면 부분과 사이드 패널과의 색 조합을 통해 도면 부분에 시선 흐름이 오래 가도록 구현하는 것에 어려움이 있었습니다.

### 6.개선 및 아쉬운 점
  도면 탐색과 비교 기능을 구현하면서 사용자가 단순히 보는 것을 넘어 판단하고 기록하는 기능도 있으면 좋겠다고 생각하였습니다.
  현장 소장의 업무 흐름을 고려하면 이전 리비전과 무엇이 달라졌는지 공종 간 충돌은 없는지 확인한 뒤 그 결과를 따로 기록하거나 공유해야 합니다.
  현재 프로토타입은 도면 비교까지는 가능하지만 비교 결과를 서비스 내부에 남길 수 있는 기능은 구현하지 못했습니다. 시간이 더 주어진다면 다음 기능을 추가하고 싶습니다.
  #### 도면 메모 기능
  특정 도면 또는 리비전에 대해 텍스트 메모를 남기고 리비전 변경 시 함께 추적할 수 있는 기록 기능
  #### 주석 / Pen 기능
  도면 위에 직접 선을 긋거나 강조 표시를 하여 변경된 부분이나 충돌 예상 지점을 시각적으로 표시하는 기능
  
   이 기능이 추가된다면 단순 도면 뷰어가 아니라 협업 및 의사결정을 지원하는 도면 관리 도구로 발전할 수 있을 것이라 생각하였습니다.
